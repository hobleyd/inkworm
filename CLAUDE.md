## Architecture

Inkworm is a paginated ePub reader built in Flutter. The core rendering pipeline has three stages:

### 1. HTML → HtmlContent (Parsing)
`EpubParser` reads the ePub zip archive, extracts chapter HTML, and walks the XML tree. Each XML node is dispatched to an `HtmlHandler` (registered by tag name in `HtmlHandler._htmlHandlers`). Handlers live in `lib/epub/handlers/` and produce `HtmlContent` objects (`TextContent`, `ImageContent`, `LinkContent`, `TableContent`, `ParagraphBreak`, `LineBreak`). CSS is parsed by `CssParser` and applied through `BlockStyle` / `ElementStyle` as content is built.

### 2. HtmlContent → Lines → Pages (Layout)
`BuildLine` accumulates `LineElement` objects (words, separators, images) into a `Line`, wrapping when the available width is exceeded. `BuildPage` accumulates `Line` objects into a `Page`, paginating when the canvas height is exceeded. `EpubChapter` drives this by calling `BuildPage.addContent()` with a chapter's parsed content. Tables are a special case: each cell's content is laid out in a temporary `BuildPage` at the cell's width, then all columns in a row are transferred to the real page together so heights can be reconciled.

### 3. Lines → Canvas (Rendering)
`PageRenderer` (a `CustomPainter`) iterates the `Line` list on a `Page` and calls `LineElement.paint()` for each element. Footnotes are painted after main body lines. Background rects (table cell colours) are drawn first.

### Isolate Architecture
Parsing is offloaded from the UI thread. `IsolateWorker` spawns a coordinator isolate which in turn creates a pool of `WorkerSlot` isolates (one per CPU core). Text and image measurement requests are sent back to the UI isolate (the only place Flutter can measure text), then returned to the worker slots. `PageSizeIsolateListener` bridges `PageSize` changes from the UI into the isolate request pipeline. The `Epub` Riverpod provider implements `IsolateListener` and drives the overall book-open lifecycle through a `BookState` bitmask.

### DI and State
- **GetIt + injectable** for singletons (`EpubParser`, `BuildPage`, `BuildLine`, `CssParser`, `PageSize`, caches). Handlers self-register in their constructors.
- **Riverpod (riverpod_annotation)** for UI-facing state: `Epub` (book data + parse lifecycle), `BookStateManagement`, `Progress`, `FontSize`, `Theme`, `Update`. Generated `.g.dart` files must be kept in sync via `build_runner`.
- **Drift/sqflite** (`ReadingDB`) stores reading progress per book.

| File | Role |
|------|------|
| `lib/epub/handlers/html_handler.dart` | Handler registry and abstract base |

| `lib/epub/handlers/table_handler.dart` | `<table>` → `TableContent` |

| `lib/epub/styles/element_style.dart` | Inline style (font, colour, drop-caps) |
| `lib/epub/styles/block_style.dart` | Block style (margins, alignment, display) |
| `lib/widgets/page_renderer.dart` | Canvas painter |
| `lib/providers/epub.dart` | Top-level Riverpod notifier; isolate listener |

### Testing
Tests use `flutter_test` with `GetIt` for manual DI setup (no auto-injection). Every test `setUp` registers singletons directly and `tearDown` calls `GetIt.instance.reset()`. Mocks are generated by `mockito` via `@GenerateMocks`; regenerate with `build_runner`. The test `WorkerSlot.staticUIPort` shim replaces the real UI port so text measurement works in the test process without a live isolate.
